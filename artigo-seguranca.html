Top 10 OWASP: vulnerabilidades que todo dev precisa conhecer

Por Rafael 15 de Março de 2025

Você já recebeu um e-mail dizendo que sua senha vazou? Ou viu uma notícia sobre um banco ter dados de clientes roubados? Por trás de quase todos esses casos existe uma falha de segurança que poderia ter sido evitada. A OWASP (Open Worldwide Application Security Project) é uma organização sem fins lucrativos que mantém uma lista das 10 vulnerabilidades mais críticas em aplicações web — o famoso OWASP Top 10.

"Proteger os dados dos usuários não é opcional — é a coisa certa a fazer."

O que é a OWASP?

A OWASP é uma comunidade aberta dedicada a melhorar a segurança de software. Fundada em 2001, ela produz ferramentas, documentação e padrões gratuitos que são referência mundial. O OWASP Top 10 é atualizado periodicamente (última versão: 2021) e serve como guia obrigatório para desenvolvedores e auditores de segurança.

As 10 principais vulnerabilidades

1. A01 — Quebra de Controle de Acesso (Broken Access Control)

Ocorre quando usuários conseguem acessar recursos ou executar ações que não deveriam ser permitidas. Exemplos: acessar dados de outro usuário alterando um ID na URL, acessar painel administrativo sem ser admin.

// VULNERÁVEL: sem verificação de permissão
app.get('/api/usuarios/:id', (req, res) => {
  const usuario = db.buscarUsuario(req.params.id);
  res.json(usuario);
});

// SEGURO: verificação de autorização
app.get('/api/usuarios/:id', autenticar, (req, res) => {
  if (req.usuario.id !== req.params.id && !req.usuario.isAdmin) {
    return res.status(403).json({ erro: 'Acesso negado' });
  }
  const usuario = db.buscarUsuario(req.params.id);
  res.json(usuario);
});

2. A02 — Falhas Criptográficas (Cryptographic Failures)

Dados sensíveis (senhas, cartões de crédito, dados pessoais) armazenados ou transmitidos sem criptografia adequada. Inclui uso de algoritmos obsoletos como MD5 ou SHA1 para senhas.

  • Sempre use HTTPS para transmissão de dados
  • Use bcrypt ou argon2 para hash de senhas, nunca MD5 ou SHA1
  • Criptografe dados sensíveis em repouso no banco de dados

3. A03 — Injeção (Injection)

Dados não confiáveis são enviados como parte de um comando ou consulta. O tipo mais conhecido é o SQL Injection, mas também inclui injeção de comandos OS, LDAP e NoSQL.

// VULNERÁVEL: SQL Injection
const query = "SELECT * FROM usuarios WHERE nome = '" + nome + "'";

// SEGURO: Consulta parametrizada
const query = "SELECT * FROM usuarios WHERE nome = $1";
db.query(query, [nome]);

4. A04 — Design Inseguro (Insecure Design)

Falhas de arquitetura e design que não podem ser corrigidas apenas com código. Inclui falta de modelagem de ameaças, ausência de princípio de menor privilégio e design que não considera cenários de abuso.

5. A05 — Configuração Insegura (Security Misconfiguration)

Configurações padrão inseguras, recursos desnecessários habilitados, mensagens de erro detalhadas expostas ao usuário, cabeçalhos HTTP de segurança ausentes.

  • Desabilite listagem de diretórios
  • Remova páginas e contas padrão
  • Configure cabeçalhos de segurança (Content-Security-Policy, X-Frame-Options)
  • Mantenha software e dependências atualizados

6. A06 — Componentes Vulneráveis (Vulnerable and Outdated Components)

Uso de bibliotecas, frameworks ou dependências com vulnerabilidades conhecidas. Ferramentas como npm audit e Snyk ajudam a detectar esses problemas.

# Verificar vulnerabilidades em projetos Node.js
npm audit

# Corrigir automaticamente
npm audit fix

7. A07 — Falhas de Identificação e Autenticação

Implementações fracas de login, sessão e autenticação. Inclui senhas fracas permitidas, falta de proteção contra brute force e armazenamento inseguro de tokens de sessão.

8. A08 — Falhas de Integridade de Software e Dados

Código e infraestrutura que não protegem contra violações de integridade. Inclui atualizações de software sem verificação de assinatura, pipelines CI/CD inseguros e deserialização não confiável.

9. A09 — Falhas de Log e Monitoramento

Sem logs adequados e monitoramento, ataques podem passar despercebidos por dias, semanas ou meses. A detecção precoce é essencial para minimizar danos.

  • Registre todas as falhas de login e erros de autorização
  • Implemente alertas para atividades suspeitas
  • Armazene logs de forma segura e imutável
  • Revise logs regularmente

10. A10 — Server-Side Request Forgery (SSRF)

Ocorre quando a aplicação faz requisições HTTP a URLs fornecidas pelo usuário sem validação. Um atacante pode usar isso para acessar serviços internos, como metadados de cloud, bancos de dados ou APIs internas.

Como se proteger: boas práticas gerais

  1. Validação de entrada — Nunca confie em dados do usuário. Valide e sanitize toda entrada
  2. Princípio do menor privilégio — Conceda apenas as permissões mínimas necessárias
  3. Defense in depth — Múltiplas camadas de proteção, não dependa de uma única barreira
  4. Atualizações constantes — Mantenha todas as dependências e frameworks atualizados
  5. Testes de segurança — Integre análise estática (SAST) e dinâmica (DAST) ao pipeline de CI/CD
  6. Code review — Revisão de código com foco em segurança
